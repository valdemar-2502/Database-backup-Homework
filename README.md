### Задание 1. Резервное копирование

### Кейс
Финансовая компания решила увеличить надёжность работы баз данных и их резервного копирования. 

Необходимо описать, какие варианты резервного копирования подходят в случаях: 

1.1. Необходимо восстанавливать данные в полном объёме за предыдущий день.

1.2. Необходимо восстанавливать данные за час до предполагаемой поломки.

1.3.* Возможен ли кейс, когда при поломке базы происходило моментальное переключение на работающую или починенную базу данных.

*Приведите ответ в свободной форме.*

---
---
### Ответ:
---
`
1.1. Восстановление данных в полном объёме за предыдущий день.

### Решение: Комбинированная стратегия: Полный бэкап  + Дифференциальный бэкап.

Как работает: Раз в неделю делается полный бэкап. Каждый день делается дифференциальный бэкап (копируются все изменения, накопившиеся с момента последнего полного бэкапа). Для восстановления данных за предыдущий день потребуется только полный бэкап и последний дифференциальный бэкап.

1.2. Восстановление данных за час до поломки.

### Решение: Полный бэкап + Резервное копирование журналов транзакций (WAL или Binary Log).

Как работает: Делается полный бэкап (например, раз в сутки). Далее настроено непрерывное архивирование журналов транзакций (или периодический их бэкап, например, каждые 15 минут). Это позволяет выполнить восстановление на любой момент времени вплоть до конкретного часа или даже минуты перед сбоем.

1.3. Моментальное переключение на работающую базу.

### Ответ: Да, такой кейс возможен. Это называется  Высокая доступность и реализуется через репликацию.

Схема: Настраивается Master-Slave репликация. Если основной сервер Master выходит из строя, система автоматического мониторинга переключает нагрузку на реплику Slave. Реплика постоянно синхронизируется с мастером, поэтому потери данных минимальны . Это называется  аварийное переключение. Резервная копия здесь — это готовая к работе замена.

### Ответ:

`
### Пример команды резервирования и восстановления (pg_dump / pg_restore).

Резервное копирование (создание дампа):
Для создания дампа базы данных mydb в пользовательском формате (сжатом, для pg_restore):

### pg_dump -U username -h localhost -F c -b -v -f mydb_backup.dump mydb
-F c: формат archive (custom). Подходит для pg_restore.

-b: включать большие объекты.

-v: подробный вывод.

-f: файл, куда сохраняем.

Восстановление базы данных:
Перед восстановлением нужно создать пустую базу данных (например, mydb_restore), в которую будет производиться импорт.

### CREATE DATABASE mydb_restore;
Затем выполнить восстановление из дампа:

### pg_restore -U username -h localhost -d mydb_restore -v mydb_backup.dump
### -d mydb_restore: имя целевой базы данных.

# Автоматизация процесса.

Ответ: Да, автоматизировать можно и нужно.

Как: С помощью планировщиков задач.

Скрипт: Пишется bash/bat скрипт, который:

Выполняет команду pg_dump (или pg_basebackup для физического бэкапа).
Проверяет код возврата (успех/ошибка).
Загружает полученный бэкап в удаленное хранилище (Яндекс.Облако, отдельный сервер).
Удаляет старые бэкапы, чтобы не забивать диск.
Инструменты: Можно использовать специализированные менеджеры бэкапов.

### Задание 3. MySQL

3.1. С помощью официальной документации приведите пример команды инкрементного резервного копирования базы данных MySQL. 

3.1.* В каких случаях использование реплики будет давать преимущество по сравнению с обычным резервным копированием?

*Приведите ответ в свободной форме.*

---
### Ответ:
---
`
Пример команды инкрементного резервного копирования.

В MySQL "чистое" инкрементное копирование делается на основе бинарных логов. Сам mysqldump инкрементные копии делать не умеет. Процесс выглядит так:

### Полный бэкап:
mysqldump -u username -p --all-databases --flush-logs --master-data=2 > full_backup.sql
--flush-logs: закрывает старый бинарный лог и начинает новый.

--master-data=2: записывает в дамп имя и позицию текущего бинарного лога.

### Инкрементный бэкап:
Инкрементным бэкапом является копирование бинарных логов, созданных после полного бэкапа.
Скопировать бинарные логи из директории данных MySQL в папку с бэкапами
cp /var/lib/mysql/mysql-bin.000001 /var/lib/mysql/mysql-bin.000002 /backup/incremental/
Для выполнения PITR (восстановления на момент времени) эти логи подаются на вход mysqlbinlog.

 Преимущества реплики перед обычным резервным копированием.

- Время восстановления из реплики измеряется секундами (просто переключить трафик). Восстановление из бэкапа может занять часы (заливка дампа).

- В реплике (при асинхронной) потери данных минимальны (несколько секунд), при синхронной — потери равны нулю. При бэкапе (если он делается раз в сутки) можно потерять до суток данных.

Нагрузка: Снятие нагрузки на чтение. Можно выполнять тяжелые SELECT-запросы или создавать отчеты на реплике, не нагружая основной сервер.

Проверка бэкапов: Реплику можно использовать для проверки, работает ли бэкап.
